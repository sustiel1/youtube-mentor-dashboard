## SECTION 2 — Detected Issues

### P1 — DUPLICATE CONTENT: `AGENTS.md` vs `docs/workflow.md`
Both files contain identical content: "Base44 + Claude Code Workflow" (117 lines).
`CLAUDE.md` also contains the same workflow section (overlap).

**Risk:** Future sessions may update one copy and miss the other, causing divergence.

**Recommendation:** Keep `AGENTS.md` at root (Claude Code convention). Delete `docs/workflow.md` or reduce to a single line pointing to `AGENTS.md`.

---

### P2 — ROOT CLUTTER (10 MD files at root)
Industry convention reserves root for: `README.md`, `CLAUDE.md`, `CHANGELOG.md`, `LICENSE.md`.

Files that don't belong at root:
- `MARKET_DASHBOARD_UI_GUIDE.md` — UI blueprint
- `CHAPTER_ENGINE_ROOT_CAUSE_REPORT.md` — debug analysis report
- `ROUTES_AUDIT_REPORT.md` — stale audit artifact
- `PROMPTS.md` — reusable prompts

**Recommendation:** Move to appropriate `docs/` subdirectories.

---

### P3 — STALE AUDIT ARTIFACTS AT ROOT
`ROUTES_AUDIT_REPORT.md` (generated 2026-06-01) and `CHAPTER_ENGINE_ROOT_CAUSE_REPORT.md` (2026-06-22) are one-time analysis outputs. They clutter the root.

**Recommendation:** Move to `docs/99_Archive/`.

---

### P4 — HIERARCHY AMBIGUITY: `AI_DEVELOPMENT_GUIDE.md` vs governance docs
`AI_DEVELOPMENT_GUIDE.md` (§1–§34) is the "original constitution." Governance docs in `docs/governance/` supersede it where they conflict (per `MASTER_PROJECT_BIBLE.md`). But the file still contains sections that may contradict newer governance docs (e.g., §24 App Ideas paths are superseded by §30, which is itself inside the same file).

**Risk:** A new Claude session reading only `AI_DEVELOPMENT_GUIDE.md` may follow superseded rules.

**Recommendation:** Add a reconciliation header at the top of `AI_DEVELOPMENT_GUIDE.md` listing which sections are superseded. Do NOT rewrite the file — section numbers (§12, §22, etc.) are cross-referenced in governance docs.

---

### P5 — `OBSIDIAN_PERSONAL_BRAIN_PHASE.md` PARTIALLY STALE
Describes ZIP export architecture. Current architecture uses direct vault API (`/api/vault/write` + merge engine). Confusing for future sessions.

**Recommendation:** Add a deprecation notice at top, pointing to `SAVE_SYSTEM_ARCHITECTURE.md` for current architecture.

## SECTION 5 — Project Brain Governance Rules (Recommended)

These rules should be added to `CLAUDE_CODE_GOVERNANCE_MODE.md`:

```
### Documentation Governance

1. Every new major feature must have an entry in PROJECT_DOCUMENTATION_INDEX.md.
2. Every architecture decision must be logged in PROJECT_DECISIONS_HISTORY.md.
3. Deprecated docs must move to docs/99_Archive/ with a deprecation header — never delete.
4. PROJECT_STATUS.md must be updated at the end of each significant session.
5. Docs must be updated BEFORE a commit when architecture changes.
6. One-time audit/debug reports belong in docs/99_Archive/ — not at root.

### Required Fields in Every MD Doc

---
**Purpose:** One line.
**Status:** current | needs-review | archive
**Last Updated:** YYYY-MM-DD
**Related Files:** [links]
**Source of Truth:** YES — authoritative for X / NO — implementation in {file} is authoritative
---
```
